# 구현 계획: context 및 max_tokens 로직 교체

**작성일:** 2026-03-19
**관련 연구:** `research_260319_v1.md`

---

## 1. 목표

기존의 부정확한 `context` 및 `max_tokens` 처리 로직을 전부 제거하고, `~/ref/opencode` 참고 방향에 맞춰 다음을 구현합니다.

- 모델별 `context` 한도와 `max output` 한도 분리 저장
- 요청 payload에 `max_tokens` 실제 반영
- 부정확한 사전 토큰 추정 제거
- 업스트림 context overflow 오류 정규화
- overflow 계열 `400` 재시도 차단

---

## 2. 작업 항목

### 2.1 기존 로직 제거

- `src/utils/tokenizer.py` 삭제
- `src/handlers/chat.py`의 기존 `check_context_length()` 호출 제거
- 기존 토큰 추정 기반 early return 제거

**상태:** ✅ Done

### 2.2 모델 한도 메타데이터 재구성

- `models.json`의 각 모델에 `max_output_tokens` 필드 추가
- 기존 `context_length`는 유지
- 현재 저장 상태 점검 결과를 기준으로 누락 모델 보강

예시:

```json
{
  "name": "ollama-cloud:minimax-m2.5",
  "model": "ollama-cloud:minimax-m2.5",
  "context_length": 128000,
  "max_output_tokens": 8192
}
```

**상태:** ✅ Done

### 2.3 모델 한도 로더 추가

- `src/utils/model_limits.py` 신규 생성
- 모델별 `context_length`, `max_output_tokens` 조회 함수 구현
- 존재하지 않는 모델에 대한 안전한 fallback 처리

**상태:** ✅ Done

### 2.4 요청 검증 로직 재구현

- `max_tokens`가 주어졌을 때만 유효성 검증
- 검증 범위:
  - 정수 여부
  - 1 이상 여부
  - 모델 `max_output_tokens` 초과 여부
- `context`는 로컬 추정으로 막지 않음

**상태:** ✅ Done

### 2.5 표준 provider payload 수정

- `src/handlers/chat.py` 표준 provider payload에 `max_tokens` 포함
- `tools`, `tool_choice`와 함께 업스트림으로 전달

**상태:** ✅ Done

### 2.6 업스트림 overflow 오류 정규화

- provider 메시지 패턴 기반 정규화
- `context_length_exceeded` 계열 응답 매핑

**상태:** ✅ Done

### 2.7 재시도 정책 수정

## 4. 완료 기준

- 더 이상 `src/utils/tokenizer.py`를 사용하지 않음
- 모델별 `context_length`와 `max_output_tokens`를 저장함
- 표준 provider 요청에 `max_tokens`가 실제로 전달됨
- `max_tokens` 초과 요청은 로컬에서 명확한 오류로 반환됨
- upstream context overflow는 정규화된 오류로 처리됨
- overflow 계열 `400`은 키를 바꿔 재시도하지 않음
